## Update workflow
1. Capture pack via `scripts/ts18-create-evidence-pack.sh`.
2. Summarise using `scripts/ts18-summarise-evidence-pack.py`.
3. Generate proposal using `scripts/ts18-propose-gap-matrix-update.py`.
4. Manually review and edit this matrix; do not auto-apply.

Tier 3 candidate drafts may be generated from failed/partial validated scenarios only; missing evidence alone must not trigger candidates.
This matrix tracks the parity status between stock TS18/TW/TWTHEME behaviour (as observed in t-music evidence, diagnostics, and public ecosystem sources) and the current Auxio-TS Tier 1 Android-standard implementation.

See canonical tier model: [`docs/TS18_INTEGRATION_ARCHITECTURE.md` — TS18 Native Parity Strategy](TS18_INTEGRATION_ARCHITECTURE.md#ts18-native-parity-strategy)

## Rules

- Do not claim hardware validation has passed unless evidence is linked.
- Use `Requires TS18 validation` for all unvalidated runtime behaviour.
- Use `No` / `Maybe` / `Yes` for "Native/private investigation needed?".
- Keep private/native production eligibility conservative; Tier 4 requires an explicit human-approved design PR.

## Column definitions

| Column | Description |
|--------|-------------|
| **Area** | Feature/behaviour domain |
| **Stock TS18 / t-music expectation** | What t-music evidence, diagnostics, or ecosystem sources suggest the TS18 platform does |

| **Remaining TS18/TWTHEME validation** | What must be confirmed on real hardware (Tier 2) |

| **Confidence** | Evidence confidence label |
| **Porting decision** | Porting decision label |
