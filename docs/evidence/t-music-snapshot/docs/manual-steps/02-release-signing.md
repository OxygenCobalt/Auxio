# Manual gate: release signing (shared UID package)

This workspace produces *unsigned* APKs via `scripts/02_build_unsigned.sh`.
For new deployment flows, preserving the original vendor APK signing certificate is **not required**. You may sign with a new debug/release key for this project.

For same-package replacement/update on a device that already has `com.tw.music` installed, the installed package signing lineage still controls whether install-over-existing succeeds.

**Package:** `com.tw.music`  
**sharedUserId:** `android.uid.system`

> Warning: because `sharedUserId=android.uid.system`, mismatched signing keys will fail install/update with signature errors.

## Preflight

1. Confirm unsigned build artifact exists:
   ```bash
   test -f dist/com.tw.music-unsigned.apk
   ```
2. Confirm package identity remains `com.tw.music` and shared UID is unchanged in manifest.
3. Confirm keystore path, alias, and passwords are available via environment variables (do not commit secrets):
   - `KEYSTORE`
   - `KS_ALIAS`
   - `KS_PASS`

## Sign and verify

```bash
export KEYSTORE=/path/to/your.keystore
export KS_ALIAS=myalias
export KS_PASS=mypassword
bash scripts/05_sign_verify.sh
```

Expected output/artifact:
- Signed APK generated by signing script.
- `apksigner verify` success in script output.

## Install-over-existing validation (critical for release)

1. Attempt install/update on target TS18 unit where `com.tw.music` already exists.
2. Verify update succeeds without uninstall (preserves app identity and shared UID behavior).
3. Launch app and validate critical flows from `docs/release-validation-matrix.md` P0 rows.

## Common failure modes and diagnostics

- `INSTALL_FAILED_UPDATE_INCOMPATIBLE` / signature mismatch:
  - Cause: signing key differs from currently installed package lineage.
  - Action: use correct keystore lineage; do not change package name or shared UID.

- `INSTALL_FAILED_SHARED_USER_INCOMPATIBLE`:
  - Cause: shared user signing set is incompatible with installed packages under `android.uid.system` constraints.
  - Action: align signing lineage with system image expectations.

- Signed artifact verifies but runtime integration fails:
  - Cause: non-signing behavioral regression.
  - Action: run release validation matrix and vendor token checks before retrying release.

## Release gate decision

Do not mark release-ready until:
- Signing verification passes,
- Install-over-existing passes on target device class,
- All P0 matrix rows in `docs/release-validation-matrix.md` pass.


## Important distinction: signing vs vendor runtime compatibility

- A new signature does **not** relax vendor runtime token compatibility requirements.
- Keep vendor-facing actions/tokens/properties stable regardless of signing choice.
- `sharedUserId=android.uid.system` remains a separate compatibility boundary and must not be changed casually.
