# De-obfuscation Guidance — com.tw.music

## Current readability state

The app was compiled with ProGuard/R8 obfuscation. The result is a codebase where:

- Application-layer class names are replaced with sequential identifiers (`C0781k`, `C0516d`, etc.)
- Package names are compressed into single-letter nesting (`p060tw`, `p011c/p015b/p016a/...`)
- Method and field names are single or double letters in many places
- Some well-known library classes (IjkMediaPlayer, AndroidX support) retain readable names

The app runs on **TS18 Android 13 head units** under the TW vendor firmware. Many of the opaque classes are vendor-bound (AIDL stubs, property readers, TW integration points) and must be documented carefully even when they cannot be fully renamed.

JADX auto-aliasing has partially recovered some names, but these aliases can be wrong or misleading. The MT-normalised output provides a second perspective.

De-obfuscation is **ongoing and part of the development scope**, not a completed or abandoned phase. Every improvement to readability makes subsequent development safer and faster — especially for vendor-bound code that must be preserved exactly.

---

## Principles

1. **Understand before renaming.** Read the smali carefully. Understand what a class/method/field does before assigning a name.
2. **Conservative, one at a time.** Rename one entity per commit. Rebuild. Verify. Then proceed.
3. **Semantic names, not guessed originals.** The original source names are not recoverable. Use names that describe observable behaviour: `handleMediaCommand` not `onReceiveMediaIntent_original`.
4. **Record everything in `mappings/manual-enigma/`.** This is the persistent, authoritative record of confirmed renames.
5. **Never invent facts.** If you are not sure what a method does, leave it unnamed and add a `# TODO: unclear` comment instead of guessing.
6. **Do not bulk rename.** Mass renames based only on JADX auto-aliases are dangerous — aliases are generated by heuristics and can be incorrect.

## Tools and their roles

### Smali — primary truth

`app/apktool/smali*/` is the ground truth. Always start here.

- Read the register flow, invoke targets, and string literals to understand behaviour.
- Check which classes a method calls to understand its context.
- Look at the `implements` and `extends` directives to understand class roles.

### JADX aliased reference — first readability pass

`reference/jadx-aliased/` — use this for a higher-level Java view. The aliased names are more readable than raw JADX output, but are heuristically generated.

- Good for: getting an overview of a class's structure, seeing call chains, understanding general flow.
- Unreliable for: exact method names (aliases may be wrong), inner class relationships (may be flattened or incorrect), exception handling (often incorrect in decompiled output).

### JADX raw reference — fallback for aliased errors

`reference/jadx-raw/` — use when aliased output seems wrong or incomplete.

- Obfuscated names (`C0781k`, `p060tw`) are present here, matching the smali more closely.
- Useful for: cross-referencing when aliased output has misidentified a method, checking exact class names that map to smali directives.

### MT-normalised reference — secondary readability aid

`reference/jadx-mt/ (optional; may be absent unless generated)` — MT Manager's normalised output provides a third perspective.

- Use to cross-check JADX interpretations.
- Sometimes MT normalisation produces clearer output for specific patterns JADX struggles with.
- Still read-only — do not edit.

### `mappings/manual-enigma/` — persistent rename record

This is the Enigma-format mapping file for all human-confirmed renames.

- Format: standard Enigma `.mapping` — `CLASS obfuscated/Name deobfName`, `METHOD ...`, `FIELD ...`
- Every confirmed rename must have an entry here.
- JADX can consume these mappings to produce a better-aliased reference.
- Do not auto-generate entries — only hand-reviewed renames belong here.

### `docs/reports/jadx-problems.txt` — JADX failure register

This file lists JADX decompile failures. For any class or method listed here:

- Do not trust the Java reconstruction.
- Read the raw smali directly.
- Add a `# NOTE: JADX decompile failed here — smali is ground truth` comment to the smali file.

## Step-by-step rename workflow

```
1. Identify a class to rename.
   - Start with classes called most frequently in readable code (MusicService, MusicActivity collaborators).
   - Or start with classes that implement a known interface (e.g. implements `Runnable`, `BroadcastReceiver`).

2. Read the smali file.
   - Note the `implements` and `extends` directives.
   - Note all public/protected methods and their signatures.
   - Note string literals and intent action strings — these are often the strongest clues.
   - Note which system API classes the methods call.

3. Cross-reference with JADX aliased output.
   - Find the corresponding file in reference/jadx-aliased/.
   - Use it to understand the overall structure, but check smali for accuracy.

4. Assign a semantic name.
   - Use the naming convention of the surrounding known classes.
   - For a class in com.tw.music.*: follow the existing package structure.
   - For a utility class: name it after its primary responsibility.

5. Add a smali header comment:
   # Renamed: MediaCommandDispatcher (was C0781k)
   # Role: Receives com.tw.music.action.* intents and dispatches to MusicService.

6. Add a mapping entry in mappings/manual-enigma/:
   CLASS com/tw/music/C0781k MediaCommandDispatcher

7. Rebuild:
   bash scripts/02_build_unsigned.sh

8. Verify build passes. Commit.
```

## When to add comments instead of renaming

Not every unclear area needs a rename. For methods or fields where the behaviour is partially understood but not fully clear:

- Add a `# purpose: <what you know>` comment in smali.
- Add a `# TODO: behaviour unclear — investigate further` comment where you cannot determine purpose.
- Document unresolved areas in `docs/deobf/enigma-notes.md` for future investigation.

This prevents premature or incorrect naming while still making the code more approachable.

---

## Vendor-bound classes — special handling

Classes that implement `com.tw.service.xt.aidl.*` interfaces or read `persist.tw.ijk*` / other `persist.tw.*` system properties are vendor-bound. These classes are part of the TS18 TW firmware integration surface. For these:

- Add a smali header comment identifying the vendor interface: `# Vendor AIDL stub: com.tw.service.xt.aidl.ITWCommandAidl`
- Add a smali header comment identifying which system property is read and what it controls: `# Reads persist.tw.ijk — selects IjkMediaPlayer vs native playback engine`
- Do not rename interface method signatures — the AIDL transaction codes must match the vendor service.
- You may rename the class itself if the rename is recorded in `mappings/manual-enigma/`.
- Do not alter the method bodies that write interface tokens or call `writeInterfaceToken`.
- Document the TWTHEME dispatch path if found — even if not renamed, add a comment identifying the theme-switching hook.
- For partially opaque vendor areas: document what you know, note what you do not know, and do not guess.

---

## Priority areas for de-obfuscation

Focus initial effort on the classes most relevant to the development goals:

1. **Media command dispatcher** — the class handling `com.tw.music.action.*` intents (likely `C0781k` in `com/tw/music/`)
2. **MusicService collaborators** — inner classes, runnables, and callbacks used by `MusicService`
3. **MediaSession wrapper** — whichever class creates and manages the `MediaSession`/`MediaSessionCompat`
4. **Notification builder** — whichever class constructs the `MediaStyle` notification
5. **Vendor AIDL stubs** — document with comments; rename class wrappers only

## JADX alias caveat

Packages such as `com.p060tw.music` / `com.p073tw.music` in JADX exports are decompiler alias artefacts and must not be treated as runtime package changes. Runtime identity remains `com.tw.music`.
